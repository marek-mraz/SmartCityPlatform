## Sealed Secrets Guide

### How It Works

Sealed Secrets is a Kubernetes controller + CLI tool that solves a common GitOps problem: **you cannot safely commit plain Kubernetes Secrets to Git** because they are only base64-encoded, not encrypted.

The flow looks like this:

```
Your plaintext password
        │
        │  kubeseal --raw (encrypts with cluster's public key)
        ▼
Encrypted string (Ag...)
        │
        │  stored in platform/secrets.yaml → committed to Git
        ▼
SealedSecret resource in Kubernetes
        │
        │  sealed-secrets-controller decrypts using its private key
        ▼
Regular Kubernetes Secret (used by your pods)
```

**Key concepts:**

- The **private key** lives only inside your cluster (in the `kube-system` namespace). Nobody outside the cluster can decrypt the secrets.
- The **public key** (`pub-cert.pem`) is used to encrypt. It is safe to share — you can encrypt secrets entirely offline without cluster access.
- Encrypted values are **scoped** to a specific `namespace` + secret `name`. An `Ag...` string encrypted for namespace `wordpress` with name `wordpress-credentials` will **not** decrypt if applied to a different namespace or under a different secret name. This prevents secret reuse across namespaces.
- Each `kubeseal --raw` call produces a **different ciphertext** even for the same input — this is normal (asymmetric encryption with random padding).

### Step A: Install kubeseal CLI

Download the `kubeseal` binary matching your architecture from the [official releases](https://github.com/bitnami-labs/sealed-secrets/releases).

**Linux ARM64** (e.g. Parallels on Apple Silicon):
```bash
wget https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.27.1/kubeseal-0.27.1-linux-arm64.tar.gz
tar -xvzf kubeseal-0.27.1-linux-arm64.tar.gz kubeseal
sudo install -m 755 kubeseal /usr/local/bin/kubeseal
```

**Linux AMD64** (standard x86 server/VM):
```bash
wget https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.27.1/kubeseal-0.27.1-linux-amd64.tar.gz
tar -xvzf kubeseal-0.27.1-linux-amd64.tar.gz kubeseal
sudo install -m 755 kubeseal /usr/local/bin/kubeseal
```

**macOS (Homebrew):**
```bash
brew install kubeseal
```

Verify the installation:
```bash
kubeseal --version
```

---

### Step B: Fetch the Public Key

Fetch the public key from the cluster so you can encrypt secrets offline without needing live cluster access each time:

```bash
export KUBECONFIG=kubeconfig
kubeseal --fetch-cert \
  --controller-name sealed-secrets-controller \
  --controller-namespace kube-system > pub-cert.pem
```

> **Tip:** Commit `pub-cert.pem` to your repo. It is a public key — it contains no sensitive information and lets teammates encrypt secrets without cluster access.

You can verify the cert was fetched correctly:
```bash
openssl x509 -in pub-cert.pem -noout -text | grep -A2 "Validity"
```

---

### Step C: Encrypt Secret Values

We provide an automated script to handle the encryption mapping directly from your environment file.

1. **Copy the environment file:**
   ```bash
   cp .secrets.env.example .secrets.env
   ```

2. **Edit `.secrets.env`:**
   Open the file and replace all `change-me-...` values with your actual strong passwords.
   *Note: This file is ignored by Git. Never commit it!*

3. **Run the sealing script:**
   ```bash
   bash scripts/seal.sh
   ```

This script reads your `.secrets.env`, encrypts each plaintext value using `kubeseal` with `pub-cert.pem` and the correct namespace/secret mapping, and safely writes the output into `platform/secrets.yaml`.

---

### Where the Encrypted Values Go

The script updates `platform/secrets.yaml` automatically. You will see values like this:

```yaml
# Example — wordpress section inside platform/secrets.yaml
sealedSecret:
  enabled: true
  encryptedData:
    mariadb-root-password: "AgBy3i4OJSWK..."
    mariadb-password: "AgCj7x2LMNOP..."
    wordpress-password: "AgDk9z3QRSTU..."
    oauth-client-secret: "AgEl1w4VWXYZ..."
```

---

### Verifying a SealedSecret Was Decrypted

After ArgoCD syncs, check that the controller successfully created the underlying Secret:

```bash
# List SealedSecrets in a namespace
kubectl get sealedsecrets -n wordpress

# Check the resulting plain Secret exists
kubectl get secret wordpress-credentials -n wordpress

# Describe the SealedSecret to see controller events/errors
kubectl describe sealedsecret wordpress-credentials -n wordpress
```

If decryption fails (wrong namespace/name scope), you'll see an event like:
```
Error: no key could decrypt secret
```

---

### Updating a Secretf

1. Generate a new encrypted string using Step C above.
2. Replace the old `Ag...` value in `platform/secrets.yaml`.
3. Commit and push to Git — ArgoCD will detect the change and apply the updated SealedSecret.
4. The `sealed-secrets-controller` automatically updates the underlying Kubernetes Secret.

> **Note:** Some pods (WordPress, Keycloak) may need a manual restart to pick up new database passwords if they don't watch for Secret changes:
> ```bash
> kubectl rollout restart deployment/wordpress -n wordpress
> kubectl rollout restart statefulset/keycloak -n keycloak
> ```

### Troubleshooting

| Problem | Likely Cause | Fix |
|---|---|---|
| `Exec format error` | Wrong binary architecture | Download the `arm64` build on Apple Silicon, `amd64` on x86 |
| `no key could decrypt secret` | Wrong `--namespace` or `--name` used during encryption | Re-encrypt with the correct flags |
| `pub-cert.pem` is empty | Controller not yet running when cert was fetched | Wait for `sealed-secrets-controller` pod to be `Running`, then re-fetch |
| Secret exists but pod can't read it | Secret name/key mismatch in pod spec | Check `kubectl describe secret` and compare with the env var `secretKeyRef` in the deployment |
